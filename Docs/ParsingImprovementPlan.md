# План улучшения парсинга нозлей (VvdNozzleParser)

Документ фиксирует результат ревью текущего алгоритма и дорожную карту его
развития. Цель — повысить надёжность извлечения на реальных VVD/PVElite-отчётах
без потери совместимости с существующим поведением (см. `Architecture.md`,
раздел «Development Rules»).

## 1. Как устроен парсинг сейчас

Вход: «сырой» текст из `pypdf.extract_text()` со вставленными маркерами
`<<<PAGE n>>>` (`Infrastructure/Pdf/PdfTextExtractor.cs`).

`VvdNozzleParser.Parse` выполняет последовательность проходов с приоритетом
источников:

```
ParseHistory → ParseNozzleList → ParseBom → ParseMawpFlanges
→ ParseNozzleLoads → ParseDetailedSections → ApplyCopyNotes → InferMissingValues
```

Все данные складываются в один `NozzleRow` на идентификатор штуцера. Приоритет
источников реализован через проверки `TextUtil.IsBlank(...)` и флаг `overwrite`.
Значения нагрузок выбираются по максимуму модуля (`PickMaxAbsValue`). Извлечение
полностью построено на регулярных выражениях.

**Сильные стороны**
- Чёткое разделение по источникам, приоритет соблюдён и задокументирован в README.
- Есть merge для пометок `(Copy of N.x)`.
- Гексагональная архитектура (порты/адаптеры) — парсер изолирован от PDF и экспорта.
- Регрессионные тесты на фикстурах (`Tests/ParserRegressionTests.cs`).

## 2. Найденные проблемы

### Критичные (надёжность данных)
1. **Зависимость от нестабильного вывода `pypdf`.** `extract_text()` не сохраняет
   геометрию таблиц; порядок токенов и пробелы «плывут» на многоколоночных
   страницах. Разбор таблиц нагрузок держится на пробелах между числами — это
   главный источник хрупкости.
2. **Нет провенанса и детекции конфликтов.** Не отслеживается, какой источник
   заполнил поле; конфликт (например `Size` из BOM ≠ из Nozzle List) не виден.
   Несовпавшая регуляркой строка теряется молча.
3. **Привязка таблицы нагрузок к id по «первому попавшемуся»** (исторический
   fallback в `FindNozzleIdNearLoadTable`) могла присвоить таблицу не тому
   штуцеру. *(Частично закрыто, см. §4.)*
4. **Выбор нагрузки по max-abs игнорирует смысл колонок.** Колонки VVD — это
   сочетания нагружений (рабочее/испытание/аварийное). Слепой максимум модуля
   может выбрать не определяющий случай и потерять знак/контекст.
5. **Смешение разделителей чисел.** `[\.,]` трактуется как десятичный и для `.`,
   и для `,`; разделителя тысяч нет — `1,234` (тысячи) станет `1.234`. Нагрузки
   в кН вполне бывают > 1000.

### Средние
6. **Мёртвый код `ParseBomLine`.** Дублировал `ParseBomComponent`, не вызывался.
   *(Удалён, см. §4.)*
7. **Хрупкий чанкинг BOM.** `BomComponentRx` опирается на формат id следующего
   компонента и литерал количества `1`.
8. **Склейка многостраничных таблиц** добавляет только следующую страницу
   (`i+1`); таблицы > 2 страниц не собираются.
9. **Конфляция колонки `Size`.** Флянец + обечайка + кольцо делят один
   `NozzleRow`; геометрия кольца перетирает `Size` внутренним диаметром.

### Мелкие
10. Регулярки пересоздавались на каждый `Parse()`. *(Вынесены в `static readonly
    Compiled`, см. §4.)*
11. Мягкий перенос `­` от извлечения PDF мог ломать токены `do=`.
    *(Нормализуется, см. §4.)*
12. `NormalizeStandard` обрабатывает только семейство EN1092.

## 3. Дорожная карта

Этапы упорядочены по соотношению «эффект/риск».

### Этап 0 — Быстрые безопасные правки ✅ (выполнено)
См. §4.

### Этап 1 — Layout-aware извлечение таблиц (наибольший эффект)
Проблема №1 решается на уровне извлечения, а не регулярок.

**Итерация 1 ✅ (выполнено, см. §5)** — реконструкция строк по координатам слов:
- `PdfPlumberReportTextExtractor : IReportTextExtractor` на `pdfplumber`
  (`page.extract_words()` с группировкой слов в строки по `top` и сортировкой по
  `x0`). Это держит строки таблиц нагрузок на одной линии с детерминированными
  пробелами — ровно то, что ожидает текущий `VvdNozzleParser`.
- Page-маркеры `<<<PAGE n>>>` сохранены → выход структурно совместим с pypdf.
- `FallbackReportTextExtractor` делает pdfplumber основным, pypdf — резервным
  (при отсутствии пакета/пустом выводе).

**Итерация 2 ✅ (выполнено, см. §6)** — структурированные таблицы:
- `page.extract_tables()` сериализуется в детерминированный блок
  `<<<TABLE>>>…<<<TABLE END>>>` с ячейками через `|` (явные границы колонок).
- Новый проход `ParseStructuredLoadTables` читает блоки по ячейкам и перетирает
  менее надёжные текстовые нагрузки (структурные ячейки — самый надёжный источник).
- Фикстура `structured_load_table.txt` с «съехавшими» колонками (в плоском тексте
  протекают чужие числа) подтверждает, что структурный путь побеждает.

### Этап 2 — Провенанс, уверенность и валидация ✅ (выполнено, см. §7)
- Параллельный словарь `Observations` на `NozzleRow` (значение + источник),
  публичный вид полей для экспортеров сохранён без изменений.
- `RowValidator`: sanity-проверки (`wt < od/2`, DN ↔ OD по таблице номиналов,
  диапазоны PN/Class, правдоподобность нагрузок, полнота полей) + детекция
  конфликтов между источниками; выставляет `Confidence` и список `Diagnostic`.
- Экспорт: подсветка ячеек с предупреждениями в XLSX + отдельный
  `*_nozzle_review.tsv` со всеми находками и уровнем уверенности.

**Осталось на будущее:**
- Полное провенанс-покрытие всех assignment-сайтов (сейчас инструментированы
  кросс-источниковые поля: Size/PressureClass/NozzleType/Standard/Material).
- Лог id, присутствующих в одном источнике, но не вытянутых в других
  (молчаливая потеря данных, проблема №2).

### Этап 3 — Гибрид с LLM-экстракцией (fallback-слой) 🟡 (скелет добавлен, не подключён)
- Детерминированный парсер — источник истины. LLM (Claude) вызывается **только**
  для секций, которые регулярки не разобрали или где сработала детекция
  конфликта/низкой уверенности.
- Слой за портом `INozzleFieldResolver`; результат LLM помечается источником с
  пониженной уверенностью и **никогда** не перетирает детерминированное значение.
- Кэшировать ответы по хэшу секции; офлайн-режим без сети деградирует до
  Этапа 0–2 без ошибок.

**Что уже есть (см. §8):** интерфейс `INozzleFieldResolver`, DTO `FieldSuggestion`,
конфиг `HybridResolverOptions` (из env), реализация `ClaudeNozzleFieldResolver`
(dependency-free, защищённая). **Не подключено** к `ExtractionService`/парсеру.

**Что осталось (вынесено в issue):** провижининг ключа `ANTHROPIC_API_KEY`,
включение `NOZZLE_EXTRACTOR_LLM`, интеграция резолвера в конвейер как fallback
для low-confidence/незаполненных полей, устойчивый JSON-парсинг, кэш, тесты,
учёт сетевой политики окружения.

### Этап 4 — Рефактор разбора компонента BOM (поддерживаемость)
- Заменить мегарегулярки токенайзером + небольшим конечным автоматом: токенизация
  строки компонента, затем присвоение полей по типу токена (`do=`, `wt=`,
  `EN …`, `1.xxxx`, `DN…`). Решает проблемы №7 и №9, упрощает тесты.
- Вынести реестр регулярок в один документированный модуль с unit-тестами на
  каждый паттерн.

### Этап 5 — Прочее
- Многостраничная склейка таблиц по детекту продолжения, а не `i+1` (№8).
- Корректный разбор разделителя тысяч в нагрузках (№5).
- Расширить `NormalizeStandard` и добавить fuzzy-сопоставление материалов/
  стандартов со справочником (борьба с шумом извлечения, №12).
- Вынести маппинг локальной нотации VVD (`Fl→Fx`, `Fc→Fy`, `Mt→Mz`) в конфиг.

## 4. Сделано в этом проходе (Этап 0)

- Удалён мёртвый `ParseBomLine` (дубликат `ParseBomComponent`).
- Горячие полнотекстовые регулярки вынесены в `static readonly` с
  `RegexOptions.Compiled` (BOM flange/nozzle/ring/component, summary нагрузок) —
  больше не пересоздаются на каждый `Parse()`.
- `FindNozzleIdNearLoadTable`: последний fallback теперь доверяет «голому» id
  только если в секции ровно один уникальный штуцер; иначе пропуск — лучше
  пусто, чем неверная привязка.
- `TextUtil.Normalize` убирает мягкий перенос `­` до схлопывания пробелов.

Регрессионные тесты не меняли поведение на существующих фикстурах (ветка
`DATA FOR NOZZLE:` в `FindNozzleIdNearLoadTable` срабатывает раньше fallback).
Сборка/тесты под .NET Framework (`build-tests.ps1`) запускаются на Windows-раннере.

## 5. Сделано в этом проходе (Этап 1, итерация 1)

- `Infrastructure/Pdf/PdfPlumberTextExtractor.cs` — Python-скрипт на `pdfplumber`,
  реконструирующий строки по координатам слов (кластеризация по `top`,
  сортировка по `x0`), с сохранением page-маркеров.
- `PdfPlumberReportTextExtractor` — адаптер порта `IReportTextExtractor`.
- `FallbackReportTextExtractor` — пробует экстракторы по порядку, возвращает
  первый непустой результат, при исключении/пустом выводе переходит к
  следующему. Подключён в `ExtractionService.CreateDefault`: pdfplumber → pypdf.
- `requirements.txt` с `pdfplumber`/`pypdf`; обновлены README и Architecture.md.
- Тест `FallbackExtractorPrefersFirstUsableResult` (без Python) проверяет
  деградацию: throw → fallback, blank → fallback, usable → primary, all-fail →
  проброс последней ошибки.

## 6. Сделано в этом проходе (Этап 1, итерация 2)

- `PdfPlumberTextExtractor`: дополнительно выводит `page.extract_tables()` как
  блоки `<<<TABLE>>>…<<<TABLE END>>>` с ячейками через `|`.
- `VvdNozzleParser.ParseStructuredLoadTables`: читает табличные блоки по ячейкам,
  сопоставляет метку строки локальному символу VVD (Fz/My/Mx/Fl/Fc/Mt →
  Fx/Fy/Fz/Mx/My/Mz), берёт max-abs из числовых ячеек и перетирает нагрузки,
  полученные текстовыми проходами. Проход срабатывает только при наличии блоков
  (на pypdf-выходе их нет → старое поведение не меняется).
- Фикстура `Tests/Fixtures/structured_load_table.txt` + тест
  `StructuredTableOverridesGarbledText`: текстовая таблица содержит «протёкшие»
  числа от сдвига колонок (888, 77, …), а итоговые значения берутся из
  структурного блока.

## 7. Сделано в этом проходе (Этап 2)

- Модель `Domain/Model/Diagnostics.cs`: `Confidence`, `Severity`, `Diagnostic`,
  `Observation`, имена источников `Source`.
- `NozzleRow`: поля `Confidence`, `Diagnostics`, `Observations` + метод `Observe`.
  Парсер фиксирует кандидатов от каждого источника для конфликтных полей.
- `Domain/Parsing/RowValidator.cs`: геометрия (`wt<od/2`), DN↔OD по таблице
  номиналов, известность PN/Class, правдоподобность нагрузок, полнота полей,
  конфликты источников; выставляет `Confidence` и `Diagnostic`-и. Проход вызван в
  конце `VvdNozzleParser.Parse` (значения не меняются).
- Экспорт: `XlsxWriter` подсвечивает ячейки с предупреждениями (жёлтая заливка);
  новый `ReviewReportWriter` пишет `*_nozzle_review.tsv`. `ExtractionService`
  логирует число находок и строк на проверку.
- Фикстура `Tests/Fixtures/validation_conflict.txt` + тест
  `ValidatorFlagsConflictsAndBadGeometry`: расхождение Nozzle List ↔ MAWP по
  size/class и `wt>=радиуса` → три находки и `Confidence=Low`, значения не
  изменены.

## 8. Сделано в этом проходе (Этап 3, скелет-опция)

- Порт `INozzleFieldResolver` + DTO `FieldSuggestion` (advisory, не перетирает
  детерминированные значения).
- `HybridResolverOptions.FromEnvironment()` — конфиг из env: `NOZZLE_EXTRACTOR_LLM`
  (вкл.), `ANTHROPIC_API_KEY`, `NOZZLE_EXTRACTOR_LLM_MODEL`. Секреты не коммитятся.
- `ClaudeNozzleFieldResolver` — обращение к Anthropic Messages API на
  `HttpWebRequest` + ручной JSON (компилируется под in-box .NET 4.0); полностью
  защищён: при любой ошибке/выключенном режиме возвращает пустой список.
- Тест `HybridResolverStaysInertUntilEnabled` — без сети проверяет, что резолвер
  молчит, пока не включён и не задан ключ.
- **Не подключено** к конвейеру. Включение — отдельной задачей (issue).
